# 06 - 복잡도

> [!info] 이 노트에서 끝내야 할 것
> 1. 시간 복잡도와 공간 복잡도를 분리해서 설명할 수 있어야 한다.
> 2. 왜 `O(n)`과 `O(log n)`의 차이가 입력이 커질수록 압도적으로 벌어지는지 말할 수 있어야 한다.
> 3. 왜 상수와 낮은 차수 항을 버리고 큰 흐름만 보는지 이해해야 한다.

## 1. 왜 복잡도를 배우는가

입문 단계에서는 종종 이런 기준으로 코드를 본다.

```text
잘 실행되면 좋은 코드 아닌가?
```

하지만 자료구조와 알고리즘에서는 이것만으로 부족하다.

- 정답은 맞는데 너무 오래 걸릴 수 있다
- 지금은 빨라도 데이터가 커지면 갑자기 버거워질 수 있다
- 메모리를 너무 많이 먹어 실제 환경에서 못 쓸 수도 있다

복잡도는 바로 이 문제를 다루는 언어다.
즉, **입력이 커질 때 알고리즘이 얼마나 버거워지는지**를 표현하는 도구다.

복잡도는 "코드가 동작한다"에서
"이 설계가 커져도 유지되는가"를 보는 단계다.

## 2. 초시계보다 더 중요한 것

초보자는 복잡도를 처음 배울 때 "실제로 몇 초 걸리나"만 떠올린다.
물론 실행 시간 측정도 중요하다. 하지만 그것만 보면 자주 속는다.

왜냐하면 실행 시간은 너무 많은 외부 요인의 영향을 받기 때문이다.

- 컴퓨터 성능
- 언어 구현
- 운영체제
- 캐시 상태
- 입력 데이터의 모양

그래서 복잡도는 "이 컴퓨터에서 0.03초" 같은 숫자보다,
**입력이 커질 때 증가하는 패턴**을 본다.

## 3. 시간 복잡도

시간 복잡도는 입력 크기 `n`이 커질수록 **연산 횟수**가 어떻게 늘어나는지 보는 개념이다.

핵심은 절대 시간 자체가 아니라 **성장률(growth rate)** 이다.

예를 들어,

- 선형 검색은 최악의 경우 거의 모든 원소를 본다 → `O(n)`
- 이진 검색은 절반씩 버린다 → `O(log n)`

즉, 입력이 두 배, 열 배, 백 배가 될 때 어떤 알고리즘이 더 버티는지를 보는 것이다.

## 4. 공간 복잡도

공간 복잡도는 입력 자체를 빼고, 알고리즘이 추가로 얼마나 메모리를 쓰는지 보는 개념이다.

예를 들어,

- 배열 하나를 그대로 훑기만 하면 추가 메모리는 거의 없다
- 복사본을 하나 더 만들면 그만큼 메모리를 더 쓴다
- 해시 테이블처럼 버킷과 노드를 따로 두면 추가 공간 비용이 생긴다

즉, 시간만 줄이면 끝이 아니라
**얼마나 빠르게, 얼마나 많은 메모리를 써서** 해결하는지도 함께 봐야 한다.

## 5. 왜 `O(n)`과 `O(log n)`이 그렇게 다른가

입문자에게는 이 두 표기가 비슷해 보일 수 있다.

```text
둘 다 n이 들어가는데 그렇게 큰 차이인가?
```

그런데 실제 차이는 매우 크다.

### 직관적 비유

- `O(n)`: 책을 한 장씩 넘겨가며 찾는다
- `O(log n)`: 책을 중간쯤 펼쳐 보고 절반을 계속 버린다

128개 중 하나를 찾는다고 해 보자.

- 선형 검색은 최악의 경우 128번 본다
- 이진 검색은 대략 7번 정도면 된다

왜냐하면

```text
128 -> 64 -> 32 -> 16 -> 8 -> 4 -> 2 -> 1
```

처럼 범위가 줄어들기 때문이다.

## 6. 큰 항만 남기는 이유

복잡도 표기에서 초보자가 가장 어색해하는 부분이 이것이다.

```text
왜 2n + 3을 그냥 O(n)이라고 쓰지?
```

이유는 복잡도가 **성장률의 큰 흐름**을 보기 때문이다.

입력이 커질수록

- `+3` 같은 상수는 영향이 거의 사라지고
- `2n`과 `n`은 둘 다 직선으로 자라며
- 결국 가장 빠르게 커지는 항이 전체를 지배한다

그래서,

- `2n + 3`은 `O(n)`
- `n^2 + 5n + 7`은 `O(n^2)`

로 본다.

### 한 문장으로 정리

> Big-O는 "정확한 실행 시간"이 아니라 "입력이 커질 때 지배적인 성장 패턴"을 본다.

## 7. 복잡도를 읽는 감각

### `O(1)`

입력 크기와 거의 무관하게 일정한 작업 수를 가진다.

예:

- 배열 한 칸 읽기
- 맨 끝 원소 하나 꺼내기

### `O(n)`

입력 크기에 비례해서 일이 늘어난다.

예:

- 배열 전체 순회
- 선형 검색
- 최대값 찾기

### `O(log n)`

입력을 계속 절반으로 줄인다.

예:

- 이진 검색

### `O(n^2)`

입력 크기가 커질수록 매우 빠르게 부담이 커진다.

예:

- 이중 반복문으로 모든 쌍 비교

이 노트의 핵심은 이 중 특히 `O(n)`과 `O(log n)`의 차이를 체감하는 데 있다.

## 8. 선형 검색과 이진 검색을 복잡도로 다시 보기

### 선형 검색

최악의 경우 마지막 원소를 찾거나 아예 못 찾을 때, 거의 모든 칸을 확인한다.

```text
1개 보면 1번
10개 보면 최대 10번
100개 보면 최대 100번
```

그래서 `O(n)`이다.

### 이진 검색

최악의 경우에도 매번 절반을 버린다.

```text
128개 -> 약 7번
1024개 -> 약 10번
1,048,576개 -> 약 20번
```

이게 바로 로그 성장이다.
입력이 엄청 커져도 비교 횟수는 천천히 늘어난다.

## 9. 비교 실험 코드

```python
from typing import Any, Sequence

def seq_search(a: Sequence, key: Any) -> tuple[int, int]:
    comparisons = 0
    i = 0

    while True:
        comparisons += 1
        if i == len(a):
            return -1, comparisons
        if a[i] == key:
            return i, comparisons
        i += 1

def bin_search(a: Sequence, key: Any) -> tuple[int, int]:
    pl = 0
    pr = len(a) - 1
    comparisons = 0

    while True:
        pc = (pl + pr) // 2
        comparisons += 1

        if a[pc] == key:
            return pc, comparisons
        if a[pc] < key:
            pl = pc + 1
        else:
            pr = pc - 1
        if pl > pr:
            break
    return -1, comparisons


a = list(range(1, 129))
key = 127

print("선형 검색:", seq_search(a, key))
print("이진 검색:", bin_search(a, key))
```

### 이 코드를 어떻게 읽어야 하나

이 코드는 원본 `ssearch_while.md`와 `bsearch.md`의 핵심 루프를 유지한 채,
비교 횟수 계수기만 덧댄 측정용 버전이다. 목적은 "정확히 몇 ms"를 재는 것이 아니다.
두 알고리즘이 **비교 횟수라는 관점에서 얼마나 다르게 자라는지**를 보이기 위한 것이다.

즉, 복잡도는 추상 개념이지만, 비교 횟수를 세어 보면 그 추상이 훨씬 구체적으로 보인다.

## 10. 왜 어떤 구현은 복잡도는 같아도 더 낫거나 나쁠 수 있나

여기서 중요한 미세한 감각이 하나 있다.

보초법과 일반 선형 검색은 둘 다 큰 틀에서는 `O(n)`이다.
그런데도 보초법을 배우는 이유는, **같은 Big-O 안에서도 구조가 더 깔끔하거나 실제 비교 수가 줄 수 있기 때문**이다.

즉, 복잡도는 아주 중요하지만 전부는 아니다.

- Big-O는 큰 방향을 말해 준다
- 상수, 구현 방식, 메모리 사용량은 실제 체감 성능을 바꾼다

좋은 엔지니어링 판단은 "표기 하나"로 끝나지 않고,
**설계 의도, 전제 조건, 구현 비용**을 함께 본다.

## 11. 최선, 평균, 최악

복잡도를 볼 때는 어느 상황을 말하는지도 중요하다.

예를 들어 선형 검색은

- 첫 칸에서 찾으면 매우 빠를 수 있다
- 마지막 칸이나 실패면 매우 느리다

그래서 보통

- 최선(best case)
- 평균(average case)
- 최악(worst case)

를 구분한다.

입문 단계에서는 보통 **최악의 경우**를 먼저 본다.
왜냐하면 시스템이 버티지 못하는 순간은 보통 최악의 상황에서 드러나기 때문이다.

## 12. 왜 이 단원이 해시 직전인가

복잡도 노트는 그냥 이론 설명이 아니다.
다음 해시 단원의 문을 열기 위한 준비다.

검색 단원에서 이미 이런 감각이 생겼다.

- 선형 검색은 단순하지만 느릴 수 있다
- 이진 검색은 빠르지만 정렬이 필요하다

그러면 다음 질문이 자연스럽게 나온다.

```text
정렬 없이도 더 빨리 찾을 수는 없나?
```

바로 그 질문이 해시법으로 이어진다.

## 13. 정렬 비용까지 같이 생각해야 한다

초보자는 종종 이렇게 생각한다.

```text
이진 검색이 O(log n)이니까 무조건 최고 아닌가?
```

하지만 현실에서는 정렬 비용을 함께 봐야 한다.

- 이미 정렬된 데이터에서 여러 번 찾는다면 이진 검색이 매우 좋다.
- 정렬되지 않은 데이터를 한 번만 찾을 거라면, 정렬부터 하는 비용이 더 클 수 있다.

즉, 알고리즘 평가는 함수 하나만 떼어서 하지 않는다.
데이터가 어떤 상태로 들어오고, 그 작업을 몇 번 반복할지까지 같이 봐야 한다.

## 14. 초보자가 자주 하는 오해

- 코드 줄 수가 적으면 더 빠르다고 생각하기
- 한 번의 실행 시간을 보고 전체 성능을 단정하기
- `O(2n)`과 `O(n)`이 완전히 다른 등급이라고 느끼기
- 정렬 비용과 검색 비용을 분리해서 보지 않기
- 시간 복잡도만 보고 공간 비용을 무시하기
