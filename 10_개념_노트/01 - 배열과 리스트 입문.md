# 01 - 배열과 리스트 입문

> [!info] 이 노트에서 끝내야 할 것
> 1. 왜 배열이 필요한지 스스로 설명할 수 있어야 한다.
> 2. 리스트와 튜플의 차이를 "성질"로 설명할 수 있어야 한다.
> 3. 인덱싱과 슬라이싱을 보고 결과를 예측할 수 있어야 한다.

## 1. 왜 이걸 배우는가

자료구조 수업은 사실 "상자 수업"이 아니다.
본질은 **문제를 다루기 좋은 형태로 데이터를 배치하는 법**을 배우는 것이다.

처음 프로그래밍을 배우면 보통 이렇게 쓴다.

```python
score1 = int(input('1번 학생의 점수를 입력하세요.: '))
score2 = int(input('2번 학생의 점수를 입력하세요.: '))
score3 = int(input('3번 학생의 점수를 입력하세요.: '))
score4 = int(input('4번 학생의 점수를 입력하세요.: '))
score5 = int(input('5번 학생의 점수를 입력하세요.: '))
```

이 도입 블록도 원본 `total.md`의 입력부를 그대로 따른 것이다.
즉, 이 노트는 처음부터 "좋은 리스트 코드"로 시작하지 않고, **배열이 없을 때 코드가 어떻게 흩어지는지**를 먼저 보여 준다.

이 코드는 5개일 때만 그럭저럭 읽힌다.
학생이 50명이라면 어떻게 될까? 코드 길이가 늘어나는 것보다 더 큰 문제는, **생각이 깨진다**는 데 있다.
합계를 구하려면 변수 이름을 하나하나 기억해야 하고, 최대값을 찾으려면 비교도 하나하나 손으로 늘어놓아야 한다.

배열은 이 문제를 바꾼다.
"점수 5개"를 "점수들의 모음 하나"로 보게 만든다.

이 지점은 "코드를 줄줄이 적는 단계"에서 "문제를 한 덩어리로 모델링하는 단계"로 넘어가는 순간이다.

## 2. 직관적 비유

### 비유 1. 출석부

학생 이름을 메모지 30장에 따로 적어 두는 것보다, 출석부 한 장에 순서대로 적어 두는 편이 훨씬 관리하기 쉽다.

- 메모지 30장: `score1`, `score2`, `score3` ...
- 출석부 한 장: 리스트

### 비유 2. 책장과 칸 번호

책장에 꽂힌 책을 찾을 때 "맨 왼쪽에서 세 번째 칸"처럼 위치로 부른다.
배열에서 인덱스도 정확히 그런 역할을 한다.

### 비유 3. 복도 구간

슬라이싱은 "3번 교실부터 7번 교실 전까지"처럼 **구간**을 잡는 일이다.
그래서 끝 번호를 포함하지 않는 것이 오히려 자연스럽다.
문과 문 사이의 경계선을 잡는다고 생각하면 이해하기 쉽다.

## 3. 배열과 리스트는 무엇이 다른가

강의에서는 "배열"이라는 말로 여러 값을 묶어 다루는 사고를 설명하고, 파이썬에서는 주로 **리스트**를 통해 그 생각을 구현한다.

즉, 이 강의 맥락에서는 다음처럼 이해하면 된다.

- 배열: 여러 값을 한 덩어리로 다루는 개념적 틀
- 리스트: 파이썬에서 그 틀을 구현하는 대표 자료형

그래서 수업에서는 "배열"과 "리스트"가 거의 같은 맥락으로 움직인다.

## 4. 리스트와 튜플

### 한 문장 정의

- 리스트: 변경 가능한 묶음
- 튜플: 변경 불가능한 묶음

### 코드로 보면

```python
list02 = [1, 2, 3]
list12 = [None] * 5

tuple04 = 1, 2, 3
tuple10 = tuple('ABC')

print('list02 =', list02)
print('list12 =', list12)
print('tuple04 =', tuple04)
print('tuple10 =', tuple10)
```

이 블록은 원본 `list_tuple.md`에서 가져온 축약 버전이다.
입문 단계에서 중요한 것은 "문법 예쁘게 쓰기"가 아니라, **같은 자료를 어떤 성질의 컨테이너에 담을 것인가**다.

- 리스트는 길이와 원소를 유연하게 운용하는 쪽에 가깝다.
- 튜플은 구조가 고정된 데이터, 즉 해석 규약이 중요한 값 묶음에 가깝다.
- `[None] * 5` 같은 초기화는 뒤 장에서 고정 길이 배열, 버퍼, 버킷 같은 구조를 읽을 때 반복해서 등장한다.

### 왜 굳이 둘로 나누나

초보자는 "둘 다 여러 값을 묶는데 왜 나눠?"라고 느낀다.
하지만 성숙한 프로그래밍에서는 "바뀌어도 되는 데이터"와 "바뀌면 안 되는 데이터"를 구분하는 것이 매우 중요하다.

예를 들어,

- 점수 목록: 앞으로 수정될 수 있다 → 리스트
- 좌표 `(x, y)`: 구조가 고정되어 있다 → 튜플

이 차이는 단순 문법 차이가 아니라 **표현 방식**과 **사용 의도**의 차이다.

## 5. 인덱싱

인덱싱은 여러 값 중 **한 칸만** 집어 드는 것이다.

```python
x = [15, 64, 7, 3.14, [32, 55], 'ABC']
for i in range(len(x)):
    print(f'x[{i}] = {x[i]}')
```

### 음수 인덱스는 왜 편한가

뒤에서 몇 번째인지 바로 말하고 싶을 때가 많기 때문이다.

- `values[-1]`: 마지막 원소
- `values[-2]`: 뒤에서 두 번째

원본 `list_element.md`는 일부러 서로 다른 자료형을 한 리스트에 넣어 보여 준다.
입문자에게는 낯설지만, 여기서 중요한 포인트는 두 가지다.

- 인덱스는 원소의 자료형과 무관하게 "위치"만 기준으로 동작한다.
- 리스트는 단순 숫자 상자가 아니라, **참조와 값을 담는 순서 컨테이너**다.

이 기능이 있으면 배열 길이를 굳이 계산하지 않고도 끝부분을 쉽게 다룰 수 있다.

## 6. 슬라이싱

슬라이싱은 **구간 전체**를 가져오는 것이다.

```python
list05 = list('ABC')
list09 = list(range(7))
list11 = list(range(3, 13, 2))

print(list05)      # ['A', 'B', 'C']
print(list09[2:5]) # [2, 3, 4]
print(list11[:3])  # [3, 5, 7]
print(list09[::-1])
```

### 왜 끝 인덱스는 포함되지 않나

이 규칙은 처음에는 어색하지만, 실제로는 매우 강력하다.

예를 들어 `values[:3]`과 `values[3:]`를 붙이면 정확히 전체가 된다.
경계가 겹치지 않고 딱 맞물린다.

즉,

```text
[처음 ~ 3 전까지] + [3부터 ~ 끝까지]
```

가 자연스럽게 나뉜다.
이 규칙은 나중에 분할 정복, 이진 검색, 문자열 처리에서 엄청 자주 등장한다.

## 7. 슬라이싱의 머릿속 그림

배열 칸을 보는 대신, **칸 사이의 경계선**을 본다고 생각하면 좋다.

```text
값:     11   22   33   44   55
경계:  0 | 1 | 2 | 3 | 4 | 5
```

`values[1:4]`는 경계선 1에서 시작해 4 전까지 가는 구간이다.
그래서 `22, 33, 44`만 포함된다.

이 그림을 머리에 넣으면 슬라이싱 실수가 크게 줄어든다.

## 8. 의사코드로 보면

```text
다섯 점수를 빈 리스트에 차례대로 넣는다
리스트 전체의 합계를 구한다
리스트 길이로 나누어 평균을 구한다
필요하면 일부 구간만 잘라 본다
```

이 의사코드는 강의 초반의 배열 입문 장면을 가장 잘 압축한다.
중요한 것은 "값 5개를 따로 다루지 않는다"는 점이다.

## 9. 실전 Python 코드

```python
print('학생 그룹 점수의 합계와 평균을 구합니다.')

score1 = int(input('1번 학생의 점수를 입력하세요.: '))
score2 = int(input('2번 학생의 점수를 입력하세요.: '))
score3 = int(input('3번 학생의 점수를 입력하세요.: '))
score4 = int(input('4번 학생의 점수를 입력하세요.: '))
score5 = int(input('5번 학생의 점수를 입력하세요.: '))

total = 0
total += score1
total += score2
total += score3
total += score4
total += score5

print(f'합계는 {total}점입니다.')
print(f'평균은 {total / 5}점입니다.')
```

이 코드는 원본 `total.md`를 그대로 가져온 것이다.
여기서 핵심은 이 코드가 "좋은 최종 코드"라서가 아니라, **왜 배열이 필요해지는지 강하게 드러내기 때문**이다.

- 변수 개수가 데이터 개수에 종속된다.
- 동일한 처리 로직을 사람이 직접 늘어놓아야 한다.
- 합계를 구하는 로직과 데이터 표현이 강하게 결합된다.

즉, 이 코드는 일부러 불편한 출발점을 보여 준다.
뒤 단원에서 리스트, 순회, 함수 추상화가 왜 등장하는지 이해하려면 이 불편함을 먼저 체감해야 한다.

## 10. 이 코드를 어떻게 읽어야 하나

### `score1`, `score2`, `score3`, ...`

값 자체보다 변수 이름이 먼저 늘어난다.
데이터 모델이 아니라 **변수 나열**로 문제를 풀고 있다는 신호다.

### `total += score1` 부터 `total += score5`

합계를 구하는 알고리즘 자체는 단순하지만, 데이터가 5개로 고정되어 있어서 같은 문장을 반복해서 써야 한다.
이 구조는 원소 수가 바뀌는 순간 곧바로 유지보수 비용으로 바뀐다.

### `total / 5`

평균 계산 역시 "학생 수"가 코드에 박혀 있다.
배열 기반 코드라면 `len(scores)`처럼 데이터가 자기 길이를 설명하게 만들 수 있지만, 이 버전은 그런 구조를 갖지 못한다.

## 11. 왜 이게 중요한가

이 노트는 단지 리스트 문법을 익히는 파트가 아니다.

- 데이터를 묶어 본다.
- 위치로 접근한다.
- 구간으로 자른다.

이 세 가지가 다음 노트들 전부의 출발점이 된다.

배열을 잘 다룬다는 것은, 사실상 **여러 문제를 하나의 공통된 언어로 보는 능력**을 얻는 것이다.

## 12. 초보자가 여기서 꼭 붙잡아야 할 감각

이 단원은 문법을 배우는 장면처럼 보이지만, 실제 핵심은 세 가지다.

1. 여러 값을 한 덩어리로 바라보는 감각
   변수 여러 개를 각각 다루지 않고, "같은 종류의 값들의 모음"으로 본다.
2. 위치를 기준으로 생각하는 감각
   값의 이름이 아니라 위치 번호로 접근한다.
3. 구간을 기준으로 생각하는 감각
   한 칸만 보는 것이 아니라, 앞부분과 뒷부분 같은 범위를 다룬다.

이 세 감각이 자리 잡아야 다음 단원에서

- 최대값 찾기
- 검색 범위 줄이기
- 배열 뒤집기
- 이진 검색의 구간 관리

가 자연스럽게 연결된다.

## 13. 이 단원이 뒤에서 다시 나오는 방식

- 인덱싱 감각은 선형 검색과 이진 검색에서 "어느 칸을 볼 것인가"로 이어진다.
- 슬라이싱의 경계 감각은 구간 관리와 범위 축소를 이해할 때 다시 등장한다.
- 여러 값을 한 덩어리로 보는 감각은 소수 나열, 해시 버킷, 연결 리스트 체인을 이해할 때 밑바탕이 된다.

즉, 배열 입문은 쉬운 문법 파트가 아니라, 뒤 단원 전체의 좌표계를 만드는 파트다.

## 14. 자주 하는 실수

- 리스트와 튜플을 괄호 모양 차이 정도로만 보기
- 슬라이싱 끝 인덱스를 포함된다고 착각하기
- 음수 인덱스를 단순 암기로만 외우기
- 배열을 "값 저장소"로만 보고, "문제 구조를 단순화하는 도구"로 보지 않기

## 15. 원본 실습 코드 커리큘럼 반영

이 노트는 원본 `chap02`의 도입 실습들을 "배열이라는 사고방식 입문"으로 다시 묶어 재구성한 것이다.

- 배열이 필요한 이유: `total.md`
- 리스트/튜플 생성과 성질: `list_tuple.md`, `list_element.md`
- 리스트 순회의 예고편: `list1.md`, `list2.md`, `list3.md`, `list4.md`
- 튜플 순회의 예고편: `tuple1.md`, `tuple2.md`, `tuple3.md`, `tuple4.md`

즉, 원본 실습에서는 생성과 순회가 분리되어 있었지만, 현재 노트는 그것을 "값을 따로 두지 말고 한 묶음으로 본다 -> 리스트와 튜플을 구분한다 -> 인덱스와 구간으로 읽는다"라는 학습 흐름으로 압축했다.

이 장은 아직 "비효율 개선"의 장이 아니다.
대신 이후 개선형 실습을 읽기 위해 필요한 전제, 즉 **데이터를 묶고 위치와 구간으로 읽는 기본 좌표계**를 세우는 장이다.
