# 07 - 해시와 체인법

> [!info] 이 노트에서 끝내야 할 것
> 1. 해시가 왜 빠르게 느껴지는지, "비교" 대신 "위치 계산"을 한다는 관점으로 설명할 수 있어야 한다.
> 2. 충돌이 왜 피할 수 없는지, 그리고 체인법이 그 문제를 어떻게 받아들이고 처리하는지 설명할 수 있어야 한다.
> 3. `add`, `search`, `remove`, `dump`를 연결 리스트 관점에서 읽고 구현 흐름을 말할 수 있어야 한다.

## 1. 왜 해시가 필요해졌는가

해시는 보통 "빠른 자료구조"라고 먼저 외우게 되지만,
원본 강의는 그렇게 접근하지 않는다.

강의는 먼저 **정렬된 배열의 장점과 단점**을 보여 준다.

- 장점: 검색은 빠르게 할 수 있다
- 단점: 중간 삽입은 불편하다

예를 들어 정렬된 배열 `[10, 20, 30, 40]`에 `25`를 넣는다고 하자.

1. 25가 들어갈 자리를 찾는다
2. `30`, `40`을 한 칸씩 뒤로 민다
3. 빈자리에 `25`를 넣는다

즉, 찾는 것은 빨라도 **넣는 순간 이동 비용**이 터진다.

### 이 장면이 중요한 이유

좋은 자료구조 수업은 "새 자료구조 이름"부터 주지 않는다.
기존 방법이 어디서 불편한지 먼저 느끼게 한다.

이건 설계 trade-off를 드러내는 방식이다.

- 정렬 유지: 검색은 좋다
- 정렬 유지: 삽입과 삭제가 부담스럽다

그래서 자연스럽게 이런 질문이 나온다.

```text
정렬 없이도 빨리 찾을 방법은 없나?
```

바로 그 질문의 답이 해시다.

## 2. 해시는 무엇을 바꾸는가

정렬 기반 검색은 보통 비교를 반복한다.

```text
이 값보다 큰가?
작은가?
왼쪽인가?
오른쪽인가?
```

해시는 이 접근을 바꾼다.
비교를 여러 번 하기보다 **저장 위치를 계산**한다.

```text
key -> 해시 함수 -> 버킷 번호
```

즉, 해시의 핵심 철학은 이렇다.

> 값을 찾기 위해 배열 전체를 헤매지 말고, 애초에 "갈 만한 칸"을 계산해서 바로 가자.

### 직관적 비유 1. 사물함 번호

학생증 번호를 넣으면 특정 사물함 번호가 계산된다고 생각해 보자.
그러면 모든 사물함을 열어볼 필요 없이, 해당 칸으로 바로 갈 수 있다.

### 직관적 비유 2. 옷 맡기는 번호표

코트를 맡길 때 "빨간 코트 찾기"처럼 생김새로 하나씩 찾는 대신
번호표를 기준으로 보관 위치를 정하면 찾기가 훨씬 빨라진다.

해시는 바로 이런 사고다.

## 3. 해시 함수는 무엇을 해야 하나

해시 함수는 키를 받아 버킷 번호로 바꾼다.

입문 단계에서는 보통 아래처럼 단순한 형태로 이해하면 된다.

```python
def hash_value(key: int, capacity: int) -> int:
    return key % capacity
```

예를 들어 `capacity = 13`이라면

- `18 % 13 = 5`
- `31 % 13 = 5`

둘 다 5번 버킷으로 간다.

여기서 초보자는 바로 묻는다.

```text
그럼 망한 거 아닌가?
```

아니다.
이것이 바로 충돌이고, 해시 자료구조는 애초에 충돌이 생긴다고 가정하고 설계된다.

## 4. 충돌은 왜 피할 수 없는가

버킷 수는 제한되어 있는데, 넣을 수 있는 키는 훨씬 많다.
그래서 서로 다른 키가 같은 버킷으로 가는 일은 자연스럽다.

이건 구현 실수라기보다 구조적 사실이다.

### 직관적 비유

우편함이 13개뿐인데 편지는 수백 통일 수 있다.
그러면 여러 편지가 같은 우편함 칸을 공유해야 하는 순간이 반드시 온다.

즉, 좋은 해시 테이블은 "충돌이 없게" 만드는 구조가 아니라
**충돌이 생겨도 무너지지 않게** 만드는 구조다.

## 5. 체인법의 아이디어

체인법은 한 버킷에 원소 하나만 두려 하지 않는다.
같은 버킷에 온 원소들을 **연결 리스트로 줄 세워서** 저장한다.

예를 들어 5번 버킷에 `18`, `31`, `44`가 몰리면,

```text
table[5] -> [18] -> [31] -> [44]
```

처럼 연결한다.

그래서 해시 테이블의 한 칸은 더 이상 값 하나가 아니라
**연결 리스트의 시작점(head)** 이 된다.

> [!tip] 핵심 관찰
> 해시 테이블 전체에서는 빠르게 버킷을 찾지만,
> 같은 버킷 안에서는 결국 선형 검색을 한다.

이 문장을 이해하면 체인법의 본질을 잡은 것이다.

## 6. Node가 왜 필요한가

원본 강의는 `체인법` 설명 중에 `Node`를 따로 떼어 설명한다.  이게 아주 중요하다.

초보자는 종종 해시 테이블만 보고 싶어 하지만,
실제로 연결 구조를 만드는 최소 단위는 **노드(Node)** 다.

```python
from __future__ import annotations

from dataclasses import dataclass


@dataclass
class Node:
    key: int
    value: str
    next: "Node | None" = None
```

여기서

- `key`: 검색 기준
- `value`: 실제 저장 값
- `next`: 다음 노드로 가는 연결

을 뜻한다.

즉, Node는 값 상자이면서 동시에 **연결의 고리**다.

## 7. ChainedHash 전체 구조

```python
from __future__ import annotations

from dataclasses import dataclass


@dataclass
class Node:
    key: int
    value: str
    next: "Node | None" = None


class ChainedHash:
    def __init__(self, capacity: int) -> None:
        self.capacity = capacity
        self.table: list[Node | None] = [None] * capacity

    def hash_value(self, key: int) -> int:
        return key % self.capacity

    def search(self, key: int) -> str | None:
        bucket = self.hash_value(key)
        node = self.table[bucket]

        while node is not None:
            if node.key == key:
                return node.value
            node = node.next

        return None

    def add(self, key: int, value: str) -> bool:
        bucket = self.hash_value(key)
        node = self.table[bucket]

        while node is not None:
            if node.key == key:
                return False
            node = node.next

        self.table[bucket] = Node(key, value, self.table[bucket])
        return True

    def remove(self, key: int) -> bool:
        bucket = self.hash_value(key)
        node = self.table[bucket]
        prev: Node | None = None

        while node is not None:
            if node.key == key:
                if prev is None:
                    self.table[bucket] = node.next
                else:
                    prev.next = node.next
                return True

            prev = node
            node = node.next

        return False

    def dump(self) -> None:
        for index, node in enumerate(self.table):
            print(f"{index:2}", end="")
            while node is not None:
                print(f" -> {node.key} ({node.value})", end="")
                node = node.next
            print()
```

## 8. `add`를 어떻게 읽어야 하나

### 의사코드

```text
버킷 번호를 계산한다
그 버킷의 연결 리스트를 따라가며 같은 key가 있는지 본다
이미 있으면 추가하지 않는다
없으면 새 노드를 맨 앞에 붙인다
```

### 왜 맨 앞에 붙이나

연결 리스트의 맨 앞 삽입은 간단하다.

```text
new.next = old_head
head = new
```

즉, 중간까지 내려가서 삽입할 필요 없이 바로 연결할 수 있다.

### 이때 유지되는 사실

> 해당 버킷 안에는 같은 key가 중복되지 않는다.

그래서 `search`나 `remove`가 훨씬 단순해진다.

## 9. `search`를 어떻게 읽어야 하나

### 의사코드

```text
버킷 번호를 계산한다
그 버킷의 첫 노드부터 끝까지 본다
같은 key를 만나면 value를 반환한다
끝까지 없으면 실패다
```

### 핵심 포인트

해시는 "한 번에 찾는 마법"이 아니다.
정확히 말하면

1. 해시 함수로 아주 좁은 후보 구간을 고르고
2. 그 안에서만 선형 검색을 한다

는 구조다.

그래서 충돌이 적을수록 평균적으로 빠르게 느껴진다.

## 10. `remove`가 어려운 이유

초보자가 가장 자주 막히는 부분이 삭제다.

이유는 삭제가 "현재 노드를 없애는 작업"처럼 보이지만,
실제로는 **연결을 다시 이어 주는 작업**이기 때문이다.

### 의사코드

```text
버킷 번호를 계산한다
현재 노드와 이전 노드를 함께 추적한다
삭제할 노드를 찾으면
첫 노드라면 head를 다음 노드로 바꾼다
중간 노드라면 prev.next를 node.next로 바꾼다
```

### 왜 `prev`가 필요한가

연결 리스트에서는 내 앞 노드를 모르면,
현재 노드를 빼낼 때 누가 나를 건너뛰어야 하는지 알 수 없다.

그래서 `prev`는 삭제의 핵심이다.

### 그림으로 보면

삭제 전:

```text
prev -> node -> node.next
```

삭제 후:

```text
prev -------> node.next
```

즉, 삭제는 "지우기"보다 **우회 연결 만들기**에 가깝다.

## 11. `dump`는 왜 중요한가

입문 단계에서 `dump()`는 사소해 보일 수 있다.
하지만 교육적으로는 매우 중요하다.

`dump()`를 찍어 보면

- 어떤 버킷이 비어 있는지
- 어떤 버킷에 충돌이 모였는지
- 체인이 실제로 어떻게 이어졌는지

를 눈으로 확인할 수 있다.

즉, `dump()`는 자료구조를 **보이지 않는 내부 상태에서 보이는 구조로 번역**해 주는 디버깅 도구다.

## 12. 원본 강의의 메뉴 프로그램

원본 영상 `3-3`은 자료구조 클래스 정의만 보여주고 끝나지 않는다.
실제로 `추가 -> 검색 -> 삭제 -> 덤프`를 반복하는 메뉴 프로그램으로 사용 장면까지 보여 준다.

```python
table = ChainedHash(13)

while True:
    choice = int(input("(1)추가 (2)삭제 (3)검색 (4)덤프 (5)종료: "))

    if choice == 1:
        key = int(input("추가할 키: "))
        value = input("값: ")
        if table.add(key, value):
            print("추가했습니다.")
        else:
            print("이미 같은 키가 있습니다.")

    elif choice == 2:
        key = int(input("삭제할 키: "))
        if table.remove(key):
            print("삭제했습니다.")
        else:
            print("해당 키가 없습니다.")

    elif choice == 3:
        key = int(input("검색할 키: "))
        result = table.search(key)
        if result is None:
            print("해당 키가 없습니다.")
        else:
            print("찾은 값:", result)

    elif choice == 4:
        table.dump()

    else:
        break
```

이 장면이 중요한 이유는 자료구조를 "정의"에서 끝내지 않고
**사용 시나리오 속 행동**으로 보게 만들기 때문이다.

## 13. 왜 해시가 빠르다고 말할 수 있나

엄밀히 말하면 체인법 해시는 최악의 경우 느려질 수 있다.
모든 키가 한 버킷으로 몰리면 결국 긴 연결 리스트 하나가 되어 선형 검색처럼 된다.

그런데도 해시를 빠르다고 하는 이유는,
보통은 키들이 여러 버킷에 퍼져 **평균적으로 짧은 체인**만 검사하면 되기 때문이다.

즉,

- 전체 자료를 다 보지 않고
- 먼저 버킷을 좁히고
- 작은 연결 리스트만 본다

는 구조 때문에 평균적으로 매우 강하다.

## 14. 왜 해시도 터질 수 있나

초보자는 "해시는 빠르다"는 말을 듣고 무조건 빠르다고 생각하기 쉽다.
하지만 다음 상황에서는 성능이 나빠질 수 있다.

- 버킷 수가 너무 적다
- 키가 한쪽 버킷에 몰린다
- 해시 함수가 키를 고르게 퍼뜨리지 못한다
- 충돌이 너무 많아 체인이 길어진다

즉, 해시는 마법이 아니다.
좋은 분산과 적절한 버킷 수가 함께 있어야 한다.

## 15. 이 단원의 핵심은 무엇인가

이 단원의 진짜 핵심은 "새 구조 하나 외우기"가 아니다.

- 기존 방법의 병목을 먼저 본다
- 그 병목을 줄이는 다른 표현 방식을 찾는다
- 새로운 구조가 또 어떤 비용을 내는지 본다

즉, 자료구조는 정답 목록이 아니라
**문제에 맞는 표현을 선택하는 설계 훈련**이라는 점을 가장 잘 보여 주는 단원이다.

## 16. 버킷 수가 왜 중요한가

체인법 해시에서는 버킷 수가 너무 적으면 충돌이 쉽게 늘어난다.

예를 들어 원소는 100개인데 버킷이 5개뿐이라면,
평균적으로 각 버킷에 많은 원소가 몰릴 가능성이 커진다.

그러면 해시 함수로 버킷을 빨리 찾더라도,
그 안에서 긴 연결 리스트를 다시 선형 검색해야 한다.

즉, 해시 성능은 단순히 코드의 좋고 나쁨만이 아니라

- 버킷 수
- 키 분포
- 해시 함수의 분산 정도

에 의해 함께 결정된다.

이 감각을 잡아야 "해시는 빠르다"는 말을 맹목적으로 외우지 않게 된다.

## 17. 자주 하는 실수

- 해시 함수가 완벽해야만 한다고 생각하기
- 충돌이 생기면 구현이 틀렸다고 오해하기
- `Node`가 왜 필요한지 이해하지 못한 채 코드만 외우기
- 삭제에서 `prev`가 왜 필요한지 놓치기
- 정렬된 배열 삽입과 해시의 trade-off를 연결하지 못하기
- `dump()`를 단순 출력으로 보고 학습 도구로 활용하지 않기
