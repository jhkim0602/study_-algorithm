# 05 - 검색 알고리즘

> [!info] 이 노트에서 끝내야 할 것
> 1. 키가 무엇인지, 왜 검색은 결국 "비교 기준 선택" 문제인지 설명할 수 있어야 한다.
> 2. 선형 검색, 보초법, 이진 검색의 핵심 아이디어와 전제를 구분할 수 있어야 한다.
> 3. 왜 어떤 경우에는 `O(n)`을 감수하고, 어떤 경우에는 정렬을 전제로 `O(log n)`을 택하는지 말할 수 있어야 한다.

## 1. 왜 검색을 따로 배우는가

초보자는 종종 검색을 너무 가볍게 본다.

```text
그냥 찾으면 되지 않나?
```

하지만 실제 프로그램의 큰 부분은 "새 값을 계산하는 일"보다 **이미 있는 값 중 원하는 것을 빨리 찾아내는 일**에 쓰인다.

예를 들어,

- 학생 정보에서 특정 학번 찾기
- 은행 기록에서 계좌 번호 찾기
- 게임에서 플레이어 ID 찾기
- 쇼핑몰에서 상품 코드 찾기

이런 일은 전부 검색이다.

그래서 검색은 단순한 문법 문제가 아니라, **데이터를 어떻게 저장했고 그 저장 방식 위에서 어떻게 접근할 것인가**를 묻는 자료구조 문제다.

검색 단원은 "값을 다룬다"에서 "값의 위치를 전략적으로 고른다"로 넘어가는 구간이다.

## 2. 먼저 키부터 이해해야 한다

검색은 아무 값이나 뒤지는 행동이 아니다.
무엇을 기준으로 같은지 판단할지 먼저 정해야 한다.
그 기준이 **키(key)** 다.

예를 들어 학생 레코드가 아래와 같다고 하자.

```python
student = {"student_id": 20260017, "name": "Kim", "score": 96}
```

이 학생을 찾는 기준은 여러 개가 될 수 있다.

- 학번으로 찾기
- 이름으로 찾기
- 점수로 찾기

즉, "검색"이라는 말 하나 안에도 이미 설계 선택이 들어 있다.

여기서 중요한 건 "데이터를 어떻게 표현하느냐"다.
같은 데이터라도 키를 무엇으로 잡느냐에 따라 검색 방식과 성능이 달라진다.

## 3. 원본 강의의 실제 흐름

이 강의는 검색을 추상적으로 띄워 설명하지 않는다.
영상에서는 다음 순서로 자연스럽게 전개된다.

```text
검색 기준(키)
-> 배열 안에서 찾기
-> 앞에서부터 차례대로 찾기
-> 같은 선형 검색을 조금 더 다듬기
-> 정렬된 배열에서는 절반씩 버리기
```

이 흐름이 좋은 이유는, 처음부터 어려운 구조로 가지 않고
"지금 내가 보고 있는 배열의 어느 칸을 다음에 볼 것인가?"라는 질문으로 축소하기 때문이다.

## 4. 배열 검색이 출발점인 이유

검색의 가장 기본형은 **배열의 어느 위치에 key가 있는지 찾는 문제**다.

예를 들어,

```python
values = [5, 9, 13, 21, 36]
key = 21
```

우리가 정말 알고 싶은 것은 "21이 있는가?"이기도 하지만,
실제로 프로그램이 필요한 것은 보통 그 값의 **위치(index)** 다.

왜냐하면 위치를 알아야

- 값을 읽을 수 있고
- 수정할 수 있고
- 삭제할 수 있고
- 그 주변 값도 함께 처리할 수 있기 때문이다.

그래서 검색 함수는 대개 참/거짓만 주기보다 인덱스를 준다.

## 5. 선형 검색

### 한 문장 요약

선형 검색은 **앞에서부터 끝까지 차례대로 본다**.

### 직관적 비유

종이 출석부에서 특정 이름을 찾는다고 생각해 보자.
정렬되어 있지 않다면 결국 첫 줄부터 내려가며 확인할 수밖에 없다.

이 방식은 느릴 수 있지만, 대신 **전제 조건이 거의 없다**.

### 의사코드

```text
배열의 맨 앞에서 시작한다
현재 원소가 key와 같은지 확인한다
같으면 그 위치를 반환한다
다르면 다음 칸으로 간다
끝까지 갔는데 없으면 실패다
```

### Python 코드

```python
def seq_search(values: list[int], key: int) -> int:
    for index, value in enumerate(values):
        if value == key:
            return index
    return -1
```

### 왜 맞는가

선형 검색은 사실 아주 정직한 알고리즘이다.

> 앞에서부터 현재 위치 직전까지는 이미 확인했고, 그 구간에는 key가 없다.

이 문장이 반복 내내 유지된다.

- 처음에는 아무것도 안 봤으니 당연히 참이다.
- 한 칸을 더 봤는데 key가 아니면, "지금까지 본 구간에는 없다"는 말이 계속 참이다.
- 같은 값을 만나면 바로 정답이다.

즉, 선형 검색은 단순해서 맞는 이유도 분명하다.

### 손으로 추적해 보기

배열이 `[6, 14, 28, 35, 42]`이고 `key = 35`라면:

| index | 보는 값 | 결과 |
| --- | --- | --- |
| 0 | 6 | 계속 |
| 1 | 14 | 계속 |
| 2 | 28 | 계속 |
| 3 | 35 | 성공 |

### 장점

- 정렬이 필요 없다
- 구현이 쉽다
- 데이터가 적으면 오히려 가장 실용적일 때도 많다

### 단점

- 최악의 경우 거의 끝까지 다 봐야 한다
- 데이터가 커질수록 비교 횟수가 빠르게 늘어난다

## 6. 보초법

보초법은 선형 검색을 완전히 다른 알고리즘으로 바꾸는 것이 아니다.
핵심 구조는 그대로 두고, **반복문의 조건 검사를 더 단순하게 만드는 기법**이다.

### 왜 굳이 이런 걸 배우나

초보자는 종종 이렇게 생각한다.

```text
결과만 같으면 됐지, 왜 이런 사소한 최적화를 배우지?
```

그런데 자료구조 수업에서는 이 감각이 매우 중요하다.
알고리즘은 거대한 아이디어만으로 좋아지는 것이 아니라,
같은 아이디어 안에서도 **불필요한 검사를 줄이는 습관**으로 좋아지기 때문이다.

### 직관적 비유

문 앞에 경비원을 세우는 느낌이다.
끝까지 갔는지 계속 확인하지 않고, "어차피 마지막에는 반드시 key를 만나게" 만들어 버린다.

### 의사코드

```text
배열 끝에 key를 임시로 붙인다
같은 값을 만날 때까지 앞으로 이동한다
멈춘 위치가 원래 배열 길이보다 작으면 성공이다
원래 길이와 같으면 원래 배열에는 없었던 것이다
```

### Python 코드

```python
def sentinel_search(values: list[int], key: int) -> int:
    copied = values.copy()
    copied.append(key)

    index = 0
    while copied[index] != key:
        index += 1

    return -1 if index == len(values) else index
```

### 왜 이 코드가 성립하나

`copied`의 마지막에는 반드시 `key`가 있다.
그러므로 반복문은 언젠가 반드시 멈춘다.
멈춘 위치가 원래 길이 안쪽이면 진짜 발견이고,
원래 길이와 같으면 마지막에 붙여 둔 가짜 key에서 멈춘 것이다.

### 교육적으로 중요한 포인트

보초법의 진짜 가치는 "엄청 빨라진다"보다,
**알고리즘의 구조를 바꾸지 않고도 조건 검사를 단순화할 수 있다**는 감각을 심어 주는 데 있다.

## 7. 이진 검색

### 한 문장 요약

이진 검색은 **정렬된 배열에서 반씩 버리며 찾는다**.

### 왜 갑자기 빨라지는가

선형 검색은 한 칸씩 전진한다.
이진 검색은 매번 탐색 범위를 절반으로 줄인다.

전화번호부를 찾는 장면을 떠올리면 된다.

- `Kim`을 찾고 싶은데 맨 앞부터 한 줄씩 보지는 않는다
- 중간쯤 펼쳐 보고, 너무 앞인지 뒤인지 판단한다
- 필요 없는 절반은 통째로 버린다

즉, 이진 검색은 "다음 칸"이 아니라 **다음 구간**을 고른다.

### 전제 조건

이진 검색은 정렬된 배열에서만 쓸 수 있다.

이 조건은 선택 사항이 아니다.
정렬이 깨져 있으면 "왼쪽 절반을 버려도 된다"는 결론 자체가 무너진다.

### 의사코드

```text
왼쪽 경계 left와 오른쪽 경계 right를 둔다
탐색 범위의 가운데 center를 계산한다
가운데 값이 key와 같으면 성공이다
가운데 값이 key보다 작으면 왼쪽 절반을 버린다
가운데 값이 key보다 크면 오른쪽 절반을 버린다
범위가 빌 때까지 반복한다
```

### Python 코드

```python
def bin_search(values: list[int], key: int) -> int:
    left = 0
    right = len(values) - 1

    while left <= right:
        center = (left + right) // 2
        if values[center] == key:
            return center
        if values[center] < key:
            left = center + 1
        else:
            right = center - 1

    return -1
```

### 왜 맞는가: 핵심 불변식

이진 검색을 깊게 이해하려면 다음 문장을 붙잡아야 한다.

> 반복문이 시작될 때마다, key가 존재한다면 반드시 `left`와 `right` 사이에 있다.

처음에는 배열 전체가 후보이므로 참이다.
가운데 값을 본 뒤,

- 가운데 값이 너무 작으면 왼쪽 절반 전체는 버려도 된다
- 가운데 값이 너무 크면 오른쪽 절반 전체는 버려도 된다

정렬되어 있기 때문에 이런 버림이 안전하다.
그래서 후보 구간이 계속 줄어들고, 결국 찾거나 실패를 확정한다.

### 손으로 추적해 보기

배열이 `[3, 8, 12, 20, 31, 44, 57]`, `key = 31`이라면:

| left | right | center | center 값 | 판단 |
| --- | --- | --- | --- | --- |
| 0 | 6 | 3 | 20 | key가 더 크다, 왼쪽 절반 버림 |
| 4 | 6 | 5 | 44 | key가 더 작다, 오른쪽 절반 버림 |
| 4 | 4 | 4 | 31 | 성공 |

여기서 중요한 것은 "31을 못 봤다"가 아니라
"31이 있을 수 없는 구간을 계속 잘라냈다"는 점이다.

## 8. 원본 강의의 입력 실습이 왜 중요한가

원본 강의는 이진 검색 함수를 보여주는 것에서 멈추지 않는다.
사용자가 오름차순으로 값을 입력하게 하여, **알고리즘의 전제를 실제 프로그램에서 지키는 방법**까지 보여 준다.

```python
def read_sorted_values(size: int) -> list[int]:
    values = [int(input("x[0]: "))]

    for index in range(1, size):
        while True:
            candidate = int(input(f"x[{index}]: "))
            if candidate > values[index - 1]:
                values.append(candidate)
                break
            print("바로 직전에 입력한 값보다 큰 값을 입력하세요.")

    return values
```

이 코드는 입문자에게 매우 중요하다.
좋은 알고리즘은 "좋은 함수 하나"로 끝나지 않는다.
**전제 조건을 유지하는 전체 프로그램 구조**가 같이 있어야 한다.

## 9. 세 방법을 한눈에 비교하기

| 방법 | 정렬 필요 | 핵심 아이디어 | 장점 | 약점 |
| --- | --- | --- | --- | --- |
| 선형 검색 | 필요 없음 | 앞에서부터 하나씩 확인 | 단순하고 범용적 | 데이터가 많으면 느림 |
| 보초법 | 필요 없음 | 끝 검사 비용을 단순화 | 선형 검색 개선 감각 습득 | 본질적으로는 여전히 선형 |
| 이진 검색 | 필요함 | 절반씩 버림 | 비교 횟수가 매우 적음 | 정렬 전제가 깨지면 사용 불가 |

## 10. 언제 무엇을 써야 하나

### 선형 검색이 좋은 경우

- 데이터가 아직 정렬되지 않았다
- 데이터 크기가 작다
- 코드 단순성이 가장 중요하다
- 한두 번만 찾으면 된다

### 이진 검색이 좋은 경우

- 데이터가 이미 정렬되어 있다
- 검색을 여러 번 반복해야 한다
- 비교 횟수를 줄이는 게 중요하다

### 사고의 핵심

여기서는 단순히 "더 빠른 알고리즘"을 외우는 게 아니다.
정렬이라는 추가 구조를 받아들이는 대신, 검색 시간을 크게 줄이는 **trade-off 감각**을 배운다.

## 11. 왜 어떤 검색은 잘 터지나

초보자가 검색 구현에서 자주 실패하는 이유는 크게 세 가지다.

### 1. 반환 기준이 흐리다

- 찾았을 때 무엇을 반환할지
- 못 찾았을 때 무엇을 반환할지

를 먼저 정하지 않으면 코드가 금방 꼬인다.

### 2. 전제를 코드에 반영하지 않는다

이진 검색은 정렬이 필요하다는 말을 이해해도,
입력 단계에서 정렬을 강제하지 않으면 실제 프로그램에서는 쉽게 깨진다.

### 3. 구간을 값이 아니라 인덱스로 관리해야 한다는 점을 놓친다

이진 검색은 "어떤 수를 볼까"가 아니라
"어느 구간이 아직 후보인가"를 관리하는 알고리즘이다.

## 12. 자주 하는 실수

- 키와 값 전체를 구분하지 못하고 비교 기준을 흐리게 잡기
- 선형 검색에서 실패 반환값을 정하지 않기
- 보초법에서 원본 배열을 직접 바꿔 버리기
- 이진 검색에서 정렬되지 않은 배열을 넣기
- `left`, `right` 갱신을 잘못해 무한 루프를 만들기
- `center`를 찾은 뒤에도 범위를 줄이지 않아 같은 칸을 계속 보기
