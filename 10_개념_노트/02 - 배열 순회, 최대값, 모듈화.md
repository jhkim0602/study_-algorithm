## 1. 왜 최대값 찾기가 중요한가

최대값 찾기는 그냥 숫자 하나를 고르는 문제가 아니다.
배열 문제의 핵심인 **스캔 패턴(scan pattern)** 을 가장 단순한 형태로 보여준다.

스캔 패턴이란,

```text
답이 될 후보를 하나 정한다
배열을 앞에서부터 끝까지 본다
더 좋은 후보를 만나면 답을 갱신한다
```

라는 사고 방식이다.

이건 단순히 최대값 함수를 외우는 파트가 아니라, 문제를 단계적으로 축소해 다루는 훈련이다.

## 2. 직관적 비유

### 비유 1. 반 대표 키 찾기

학생들을 한 줄로 세워 두고 "가장 키 큰 사람"을 찾는다고 해 보자.

1. 맨 앞 학생을 현재 1등으로 둔다.
2. 다음 학생을 볼 때마다 현재 1등보다 큰지 비교한다.
3. 더 크면 1등을 바꾼다.

끝까지 가면 마지막 1등이 전체 최고다.

이것이 최대값 찾기다.

### 비유 2. 경기 기록판

현재 최고 기록을 적어 둔 전광판이 있다고 생각하면 쉽다.
새 선수가 들어올 때마다 기록을 비교하고, 더 좋으면 전광판을 갱신한다.

## 3. 왜 `a[0]`을 초기값으로 두는가

초보자는 종종 이렇게 쓴다.

```python
maximum = 0
```

하지만 배열이 `[-3, -10, -1]`이면 이 코드는 틀린다.
실제 최대값은 `-1`인데, `0`이라는 가짜 후보가 이미 더 크게 들어와 있기 때문이다.

그래서 보통 첫 원소를 초기값으로 둔다.

```python
maximum = values[0]
```

이 방식은 "배열 안의 실제 값"만 후보로 삼는다는 점에서 안전하다.

## 4. 불변식으로 보면 왜 맞는가

깊이 있는 알고리즘 설명에서는 "매 단계에서 무엇이 항상 참인가"를 보는 방식이 중요하다.

최대값 탐색의 핵심 불변식은 이렇다.

> 배열의 앞부분을 이미 확인했다면, `maximum`에는 그 앞부분의 최대값이 들어 있다.

처음에는 앞부분이 첫 원소 하나뿐이므로 맞다.
그 다음 원소를 볼 때,

- 더 크면 `maximum`을 바꿔도 여전히 맞고
- 더 작으면 그대로 두어도 여전히 맞다

이 과정을 끝까지 반복하면, 마지막에는 배열 전체에 대해 불변식이 성립한다.
즉, `maximum`이 진짜 최대값이다.

## 5. 최대값 찾기 의사코드

```text
배열이 비어 있으면 오류로 본다
첫 번째 원소를 현재 최대값으로 둔다
두 번째 원소부터 끝까지 반복한다
현재 원소가 최대값보다 크면 최대값을 바꾼다
반복이 끝나면 최대값을 반환한다
```

## 6. 최대값 찾기 Python 코드

```python
from typing import Any, Sequence

def max_of(a: Sequence) -> Any:
    """시퀀스형 a 요소의 최댓값을 반환"""
    maximum = a[0]
    for i in range(1, len(a)):
         if a[i] > maximum:
            maximum = a[i]
    return maximum
```

이 코드는 원본 `max.md`의 핵심 함수다.
실전 코드 관점에서 보면 비어 있는 시퀀스 예외 처리가 빠져 있고 타입도 느슨하다.
하지만 교육적으로는 오히려 장점이 있다. **알고리즘의 뼈대만 분리해서 보여 주기 때문**이다.

- 시작 후보를 첫 원소로 둔다.
- 1번 인덱스부터 끝까지 훑는다.
- 더 큰 값을 만나면 후보를 교체한다.

문제 해결의 최소 핵심이 코드 한 덩어리로 드러난다.

## 7. 실제로 한 줄씩 추적해 보기

배열이 `[15, 2, 64, 7]`라고 하자.

| 본 원소 범위 | 현재 원소 | maximum |
| --- | --- | --- |
| 시작 | 15 | 15 |
| 2번째 원소 비교 | 2 | 15 |
| 3번째 원소 비교 | 64 | 64 |
| 4번째 원소 비교 | 7 | 64 |

이 표를 보면 "배열을 한 번 훑으며 답을 갱신한다"는 의미가 훨씬 또렷해진다.

## 8. 입력 루프와 종료 신호

강의 후반은 최대값 함수만 보여주고 끝나지 않는다.
"사용자가 그만할 때까지 입력받아 배열을 만드는 프로그램"도 함께 다룬다.

이 지점은 중요하다. 자료구조는 보통 고정된 예제 배열이 아니라 **입력으로 만들어진 배열** 위에서 동작하기 때문이다.

### 의사코드

```text
빈 리스트를 만든다
계속 입력을 받는다
입력이 End이면 반복을 끝낸다
그렇지 않으면 정수로 바꿔 리스트에 넣는다
```

### Python 코드

```python
from max import max_of

print('배열의 최댓값을 구합니다.')
print('주의: "End"를 입력하면 종료합니다.')

number = 0
x = []

while True:
    s = input(f'x[{number}]를 입력하세요.: ')
    if s == 'End':
        break
    x.append(int(s))
    number += 1

print(f'{number}개를 입력했습니다.')
print(f'최댓값은 {max_of(x)}입니다.')
```

### 왜 `while True`가 오히려 읽기 쉬운가

끝 조건이 "미리 몇 번"이 아니라 "특정 입력이 들어왔을 때"이기 때문이다.
즉, 반복 횟수보다 종료 사건이 중요할 때는 `while True + break`가 더 자연스럽다.

## 9. 모듈화는 왜 필요한가

`max_of()`를 함수로 분리하면, 입력 방식이 바뀌어도 최대값 계산 로직은 그대로 쓸 수 있다.
즉, **문제 해결 방법을 재사용 가능하게 만든다**.

- 설계(design)와 구현(implementation)을 구분하라
- 문제 해결 로직을 입력/출력 코드와 섞지 마라

모듈화는 바로 이 관점의 첫 연습이다.

### 모듈화된 구조를 의사코드로 보면

```text
max_of.py에는 최대값 함수만 둔다
run_max_of.py에서는 입력을 받고, max_of 함수를 호출한다
```

### Python 예시

```python
# max_of.py
from typing import Any, Sequence

def max_of(a: Sequence) -> Any:
    """시퀀스형 a 요소의 최댓값을 반환"""
    maximum = a[0]
    for i in range(1, len(a)):
        if a[i] > maximum:
            maximum = a[i]
    return maximum
```

```python
# max_of_test_input.py
from max_of import max_of

number = 0
x = []

while True:
    s = input(f'x[{number}]를 입력하세요.: ')

        break
    x.append(int(s))
    number += 1

print(f'최댓값은 {max_of(x)}입니다.')
```

## 10. 주석과 자료형 힌트는 왜 배우는가

원본 강의에는 `주석과 자료형 힌트`가 별도 꼭지로 나온다.
이건 단지 "예쁘게 쓰는 법"이 아니라, 코드를 더 **읽기 쉬운 구조물**로 만드는 기술이다.

### 자료형 힌트

자료형 힌트는 함수가 어떤 입력을 받고 어떤 결과를 돌려주는지 미리 알려준다.

```python
from typing import Any, Sequence

def max_of(a: Sequence) -> Any:
    """시퀀스형 a 요소의 최댓값을 반환"""
    maximum = a[0]
    for i in range(1, len(a)):
        if a[i] > maximum:
            maximum = a[i]
    return maximum
```

원본 실습은 `Sequence`와 `Any`를 사용해 함수가 특정 리스트 타입에 묶이지 않는다는 점을 드러낸다.
이는 입문용 코드라도 인터페이스를 먼저 생각해야 한다는 메시지를 담고 있다.

### 주석

좋은 주석은 **보이는 코드를 다시 읽어주는 것**이 아니라, **왜 그렇게 했는지**를 알려준다.

나쁜 주석:

```python
maximum = values[0]  # maximum에 values[0]을 저장
```

좋은 주석:

```python
maximum = values[0]  # 음수 배열도 올바르게 처리하려고 첫 원소를 시작값으로 둔다
```

## 11. 모듈 테스트는 왜 따로 하나

원본 강의의 `모듈 테스트하기` 구간은 함수 파일을 만들고 끝내지 않는다.
그 함수를 **다른 프로그램에서 불러와 실제로 써보는 과정**까지 보여 준다.

이 단계가 중요한 이유는,

- 함수가 혼자서 올바른지
- 입력 코드와 붙여도 의미가 유지되는지
- 다시 써도 설계가 깔끔한지

를 확인하게 해 주기 때문이다.

### 원본 테스트 코드 예시

```python
from max import max_of

t = (4, 7, 5.6, 2, 3.14, 1)
s = 'string'
a = ['DTS', 'AAC', 'FLAC']

print(f'{t}의 최댓값은 {max_of(t)}입니다.')
print(f'{s}의 최댓값은 {max_of(s)}입니다.')
print(f'{a}의 최댓값은 {max_of(a)}입니다.')
```

이 코드는 원본 `max_of_test.md`다.
같은 `max_of()`가 리스트뿐 아니라 튜플, 문자열, 문자열 리스트에도 동작한다는 점을 보여 준다.
즉, 함수의 진짜 가치는 특정 입력 예제 하나를 통과하는 데 있지 않고, **공통 인터페이스 위에서 여러 시퀀스를 처리할 수 있는 일반성**에 있다.

## 12. 이 노트의 핵심은 무엇인가

이 노트는 사실 다음 세 가지를 한 번에 훈련시킨다.

1. 문제를 부분으로 분해하기
   최대값 계산 로직과 입력 로직을 나눈다.
2. 추상화하기
   "점수 4개"가 아니라 "정수 시퀀스"를 다룬다.
3. 설계와 구현을 분리하기
   함수 인터페이스를 먼저 세우고, 안의 코드를 채운다.

## 13. 반례로 더 깊게 이해하기

### 반례 1. `maximum = 0`

이 코드는 양수 데이터만 보면 그럴듯해 보이지만, 음수만 있는 배열에서는 바로 깨진다.

```python
x = [-8, -3, -11]
```

이 경우 진짜 최대값은 `-3`이다.
하지만 `maximum = 0`으로 시작하면, 배열 안에 존재하지도 않는 값을 답 후보로 쓰게 된다.

즉, 초기값은 "편한 숫자"가 아니라 **문제 안에서 허용되는 실제 값**이어야 한다.

### 반례 2. 함수와 입출력을 섞어 두기

```python
def max_of():
    count = int(input("몇 개? "))
    ...
```

이렇게 쓰면 함수가

- 입력도 하고
- 계산도 하고
- 출력까지 책임질 가능성이 커진다

결과적으로 재사용이 어려워진다.
함수의 핵심 책임은 "최대값 계산"이어야 한다.

### 반례 3. 주석이 코드를 다시 읽어 주기만 하는 경우

```python
index += 1  # index를 1 증가
```

이런 주석은 초보자에게도 큰 도움이 되지 않는다.
왜 그렇게 했는지를 남겨야 한다.

## 14. 자주 하는 실수

- 빈 배열 처리 없이 바로 `values[0]` 접근하기
- `for value in values:`라고 해 놓고 첫 원소를 두 번 비교하기

- 주석에 `무엇을 하는지`만 적고 `왜 이렇게 했는지`를 남기지 않기

- 리스트 순회 패턴: `list1.md`, `list2.md`, `list3.md`, `list4.md`
- 튜플 순회 패턴: `tuple1.md`, `tuple2.md`, `tuple3.md`, `tuple4.md`
- 최대값 함수의 핵심: `max.md`
- 입력 방식별 최대값 테스트: `max_of_test_input.md`, `max_of_test_randint.md`, `max_of_test.md`
- 원소 갱신과 참조 감각: `pass_list.md`

원본 실습은 "순회 -> 최대값 -> 입력 버전 차이" 순으로 흘러가지만, 현재 노트는 이를 "배열을 한 번 훑는 스캔 패턴 -> 최대값 불변식 -> 입력과 계산 분리 -> 모듈화"라는 설계 중심 흐름으로 재배열했다.

여기서의 개선은 아직 성능 최적화보다 **표현 개선**에 가깝다.

- 값 여러 개를 순회 패턴 하나로 통일한다
- 최대값 계산을 함수로 분리해 재사용 가능하게 만든다
- 입력 방식이 바뀌어도 핵심 로직은 유지되게 만든다

즉, 뒤 장의 본격적인 알고리즘 개선에 앞서, **좋은 풀이 구조가 무엇인지**를 먼저 훈련하는 단계다.
